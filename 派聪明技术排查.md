### 讲讲你对RAG技术，RAG解决了哪些问题？
- 简单来说，RAG技术是将信息检索和文本生成模型相结合的技术框架，它让大模型回答前先查一些和问题有关的前置知识再回答，避免模型幻觉
  - 解决了这几个痛点：
    - 1.RAG强制大模型通过检索到的、可信的知识库来生成答案，减少了模型幻觉
    - 2.大模型对个人研究领域的知识不了解，RAG可以将自己领域的文献建立一个知识库，让大模型更懂用户

#### Q1.为什么使用Apache Tika对文件进行解析？
- Apache Tika是原生Java库，可以直接引入依赖进行调用，维护成本低
- Apache Tika是本地运行的开源工具，相比于第三方解析API，数据安全性更强
- Apache Tika支持多文档类型的解析，比如PDF、Word、Excel等，但对于复杂表格和双栏PDF的解析效果一般，后续打算补充专业的版面分析模型来解决问题

### 14.哪些数据需要被Redis缓存，设计缓存的原则是什么
- 分片状态这种需要实时记录文件已经上传的状态，需要放入缓存，利用Bitmap进行记录
- 高频读取的通用性数据
- 计算成本较高的结果类数据
- 设计的原则：
  - 高频读、低频写的数据，采用cache-aside策略
  - 计算昂贵的数据设置一个较长的过期时间
  - 然后就是缓存穿透、缓存雪崩、缓存击穿的应对策略
    - 当用户查询一个不存在的数据时，策略是缓存空值，并设置一个短的ttl
    - 为缓存的ttl增加一个随机值，避免集体失效导致缓存雪崩
    - 对于热key引入Redission分布式锁，只允许第一个请求去查询数据库并重建缓存，其他请求则等待或返回一个稍旧的数据，防止缓存击穿

### 5.一个带JWT的请求过来时，后端是如何进行认证和授权的？在Spring Security中，是如何集成JWT校验逻辑？
- 角色权限控制：
```
实现了一个自定义过滤器，保证每次请求执行一次认证逻辑，过滤器会从请求头的Authorization字段提取JWT，
然后对Token进行签名校验和过期检查，确保其合法性。
如果Token有效，解析出用户基本信息，然后从MySQL加载完整用户信息，封装成UserDetails对象，再新建一个认证对象，将用户身份与权限信息注入其中。
之后将认证对象存入SecurityContextHolder,也就是SpringSecurity的安全上下文中。
一旦过滤器完成了认证并设置了安全上下文，SpringSecurity就会生效，
下一次请求过来，SpringSecurity就从安全上下文中获取当前用户的角色，判断其权限
```
- 组织标签授权机制
```
通过另一个自定义过滤器实现，位于JWT认证过滤器之后，用于处理基于组织标签的细粒度数据访问控制
```
### 6.普通用户登录后，去访问一个需要特定组织标签才能查看的文件，整个后端处理的全链路流程是怎样的？
请求过来后，会由SpringSecurity的过滤器链接管，执行JWT鉴权，解析出用户信息，包装为Authorization对象注入到SpringSecurity的安全上下文，然后根据SecurityConfig中配置的接口访问规则进行基础权限校验，然后经过组织标签权限过滤器，它会从数据库中加载当前用户的组织标签信息，保存到请求上下文中，最后系统会构建一条带有权限过滤条件的ES查询（带权限过滤的ES查询是怎样的?）
