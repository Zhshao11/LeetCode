## 文献阅读AI智能助手项目
### 架构设计与技术选型
#### 1. 介绍一下你的文献阅读AI智能助手项目
- 系统遵循文件处理，知识向量化，知识检索加上答案生成流程：前端进行文件分片后，传来MD5码，后端接收文件分片，并使用Redis记录分片上传状态，然后使用MySQL进行流程记录，分片传递完成后调用MinIO的分片合成API进行合成与存储，并将上传成功的消息传给Kafka，文档解析访问监听到上传成功的消息后，使用Apache Tika对文档进行进行，并使用分块策略对文档进行分块，然后给向量化API生成向量存储到ElasticSearch里，当要查询某个问题时，会先通过ElasticSearch进行混合检索，然后把原始问题与检索结果合成到Prompt里喂给大模型，大模型生成答案流式响应给用户。
#### 2. 你在设计这个系统的整体架构时，为什么选择 Spring Boot、Spring Security、Spring Data JPA 这些技术栈？它们在这个项目中各自起到了什么关键作用？
- 首先SpringBoot作为架构的基石，它能快速整合项目中用到的一些技术栈，简化配置，让我能更专注到核心业务上。文献阅读中有些是涉及到一些企业的内部文档，是需要保证数据的安全性的，结合 JWT 生成的用户凭证，我通过 SpringSecurity 的拦截器，对每个接口进行权限校验，精准匹配用户角色、组织归属和文件的组织标签，比如普通用户无法访问其他组织的内部文献，真正落地了多级权限控制，避免越级访问，保障敏感文献数据安全。项目中需要频繁操作文件元数据、用户信息、组织标签等数据，JPA 简化了这些数据的 CRUD 操作，不用编写复杂的 SQL 语句，大幅提升了开发效率
#### 3.请详细说明你是如何设计模块化的分层架构的，各个层的职责分别是什么？如何实现功能模块间的低耦合高内聚？
- 模块的分层可以看成MVC三层：
- Model是数据处理与业务逻辑层，包括文档解析也就是大文件分片解析、调用豆包Embedding进行向量化、以及RAG架构的检索与问答逻辑实现AI交互、还包括MySQL存用户/组织/文件等元数据、ES存向量与文献检索索引、MINIO存文献文件、Redis存缓存文件上传状态和会话上下文）。
- Controller负责请求分发与流程协调，负责接收前端请求，进行请求分发与流程协调，比如接收文件上传请求、流式对话请求、权限校验请求，调用Model层进行业务逻辑处理，最终将AI检索结果，文件处理状态等响应给View，同时配合SpringSecurity做接口权限拦截。
- View：用户界面展示与交互，包括前端的用户交互界面，文献上传入口，问答对话框等，同时展示后端返回的结果。
- 功能模块的低耦合高内聚是通过Kafka中间件来实现的，具体来说，用户完成文档上传完成后，Controller会给Kafka发一个文档待向量化处理的消息，然后就可以及时响应用户上传成功，不需要等待向量化、索引构建等耗时时间，后续由Model的消费者异步监听Kafka消息，实现了View和Model层解耦。也通过消息队列的削峰填谷的缓冲作用避免用户请求阻塞。同时各功能模块各自独立，仅通过接口或消息通信，不直接依赖，保证高内聚、低耦合。
#### 4. 为什么选择 MySQL+Redis+MinIO+Elasticsearch 的多级存储结构？这种存储结构相比单一存储有什么优势？在这个项目中，每种存储分别用来存储什么数据？
- 因为项目中存在结构化数据、非结构化大文件、高维向量数据等多种数据，使用单一存储无法兼顾性能、扩展性和成本
- MinIO适合存放静态的非结构化大文件，在系统中，用户的PDF，word等原始文献，全部存在MinIO中，相比于存数据库，避免了数据库I/O成为瓶颈
- MySQL负责存储系统的核心业务逻辑数据和结构化元数据，比如用户的基本信息、组织标签、文献的文件名、上传时间、在MinIO的物理路径映射等
- Redis存分片上传记录，高频读取的通用性数据，计算结果较高的结果数据
- ElasticSearch存文档切片后进行向量化的高维向量数据。
- 这种多维数据结构不仅有效解决了数据访问控制与检索性能问题，也是系统能支撑TB级文档存储和毫秒级检索的保障。
#### 5. 你在设计基于组织标签的权限模型时，具体是如何实现的？如何结合用户角色、组织归属和文件属性进行权限过滤？
- 整个权限控制流程以 SpringSecurity 过滤器链为核心，先完成 JWT 基础鉴权，再通过自定义的组织标签过滤器做精细化权限校验，同时结合 ES 中存储的元数据，实现‘用户身份 + 组织权限 + 数据级别’的三重权限管控。
  请求到达后，首先进入 SpringSecurity 的过滤器链，由 JWT 认证过滤器解析请求头中的 JWT 令牌：先校验令牌的有效性（签名、过期时间），解析出用户 ID、角色等核心信息，再将其包装为Authentication对象注入 SpringSecurity 安全上下文。这一步不仅完成了用户身份的确认，也为后续的精细化权限校验提供了身份基础。
  接着执行自定义的组织标签权限过滤器 —— 从安全上下文获取用户身份后，关联数据库查询该用户的组织标签、可见性权限范围（如可访问哪些组织、哪些级别的文献），并将这些权限信息存入请求上下文。随后会对比目标接口 / 资源的权限规则，若用户无对应组织权限，直接抛出权限异常、拦截请求，从源头避免非法访问。
  而在文档向量化存入 ES 时，除了文本 Embedding 向量，我还将组织标签、可见性级别作为核心元数据同步存入 ES 索引。这一设计的关键价值在于：后续用户检索 / 访问文献时，ES 可直接基于元数据做权限过滤（比如只返回用户所属组织、且可见性级别为公开 / 授权的文献），无需额外查询数据库，既保证了检索效率，又实现了‘数据存储 - 权限控制’的一体化。
### 项目实现与性能优化
#### 1. 请详细说明大文件分片上传的实现流程，如何结合 Redis 缓存文件状态？断点续传是如何实现的？
- 前端将大文件按 5MB 大小分片后，会计算整个文件的 MD5 值（用于唯一标识文件，避免重复上传）和每个分片的 MD5 值（用于校验分片上传的完整性，防止分片传输过程中丢失或损坏），并将这两个 MD5 值、分片序号、总分片数一同传递给后端。
- 后端接收分片后，先校验分片 MD5 的有效性，校验通过后将分片上传至 MinIO；同时利用 Redis 的 BitMap 存储分片上传状态 —— 以文件 MD5 作为 BitMap 的 key，分片序号作为偏移量，上传成功则将对应位设为 1，未上传设为 0
- 当后端通过 Redis BitMap 判断所有分片均上传完成后，调用 MinIO 的 composeObject 接口，将所有分片按序号顺序合并为完整文件，生成最终的文件存储路径
- 断点续传的核心依赖 Redis BitMap 的状态记录：当用户上传中断（如网络异常），再次上传时，前端先请求后端查询该文件的分片上传状态（通过文件 MD5 查询 Redis BitMap），后端返回未上传的分片序号，前端仅重新上传未完成的分片，无需重复上传已成功的分片，实现断点续传。
- 文件合并完成后，后端向 Kafka 发送‘文件上传完成’的消息，通知后续业务模块（如文档解析、向量化处理模块）异步执行后续操作，避免阻塞用户请求，同时实现上传模块与处理模块的解耦。
- 文件合并完成后，后端向 Kafka 发送‘文件上传完成’的消息，通知后续业务模块（如文档解析、向量化处理模块）异步执行后续操作，避免阻塞用户请求，同时实现上传模块与处理模块的解耦。
#### 2. 你是如何进行文件的向量化处理的？使用的 Embedding API 是哪个？向量化的过程中遇到了什么问题，如何解决的？
- 首先会对文件做预处理，用项目里的 Apache Tika 工具把 PDF、Word 等格式的文献解析成纯文本，再按照豆包 Embedding API 的 token 长度限制，把长文本拆分成合适的文本块，避免因文本过长导致 API 调用失败。向量化使用的是豆包 Embedding API，调用过程中遇到过 API 超时的问题，我基于 Spring Retry 框架设计了 API 超时重试服务：设置最大重试 3 次，同时采用指数退避策略，每次重试间隔依次为 1s、2s、4s，既避免频繁请求给服务端造成压力，也提升了向量化的成功率；如果 3 次重试后仍超时，会把该文本块标记为处理失败存入 Redis 失败队列，后续通过定时任务重新处理，同时记录日志通知管理员，避免遗漏处理。向量化成功后，我会把生成的文本向量，结合文献的组织标签、文本块位置等元数据一起存入 Elasticsearch，为后续的语义检索提供支撑。

#### 4. 系统支持 TB 级文档存储和毫秒级检索，你是如何保证在大量数据下的检索性能的？Elasticsearch 的索引是如何设计的？
- 系统的文档存储使用的是MinIO，MinIO适合大文件，多并发的场景，而且支持高并发读写。
- ElasticSearch 的索引名为 knowledge_base，会存三个字段：内容、向量、权限。
- 首先是内容字段，我们定义为 textContent，它的类型是 text，底层配的是 ik 中文分词器。这个字段用来支持关键词检索的，像 BM25、match 查询都是作用在这个字段上。
- 然后是向量字段，名叫 vector，我们定义为 dense_vector 类型，维度是 2048，这个维度要和 embedding 模型匹配。这个字段用来做语义检索。当用户发起查询时，我们会把查询的内容转成向量，再用 KNN 算法去比相似度。
- 接着是权限相关的字段，在做 B 端多租户场景时要非常重视的一块。主要有三个字段：userId、orgTag 和 isPublic。
userId 标记这个文档片段属于哪个用户，用于实现私有文档的权限隔离；
orgTag 表示文档所属的组织标签，用来控制组织内的授权访问；
isPublic 标记文档是否对所有人公开。
## 难点与亮点
#### 1. 在实现多级权限控制时，遇到了什么难点？如何解决的？有没有出现过权限控制漏洞的情况？
我在实现多级权限控制时遇到的核心难点是组织标签权限和 ES 检索的联动问题：一开始我是在 ES 检索完成后，在业务层做组织标签的权限过滤，这样会导致检索出大量不符合权限的结果后再过滤，不仅性能差（检索响应时间多了 300ms 左右），还存在权限漏洞。把组织标签和可见性级别作为元数据存入 ES 的索引中，在 ES 检索阶段就通过bool查询先过滤出当前用户组织下的文献，从源头减少无效数据的返回；
#### 2. 采用 Kafka 解耦实现文件异步处理，具体是如何设计消息队列的？消息的生产和消费是如何实现的？如何保证消息的可靠性？
队列设计：创建了名为doc-processing-topic的 Topic，根据服务器的 CPU 核数设置了 2 个分区、2 个副本，保证消息的高可用性，同时避免单分区的性能瓶颈；
消息生产：在文件上传完成并合并后，用 SpringKafka 的KafkaTemplate发送消息，消息体包含文件的 MD5、MinIO 存储路径、用户组织标签等信息，发送时设置acks=all，保证消息必须写入所有副本才会返回成功；
消息消费：用@KafkaListener监听 Topic，消费端设置ack-mode=manual，只有当文件向量化、存入 ES 完成后，才手动提交 offset；如果消费失败（比如 Embedding API 超时），会把消息发送到死信队列doc-processing-dlq，后续通过定时任务重试 3 次，重试失败则通知管理员处理；
可靠性保证：
生产端开启了 Kafka 事务，保证消息发送和数据库状态更新的原子性；
消费端做了幂等性处理：用文件 MD5 作为唯一标识，消费前先检查 Redis 中是否已经处理过该文件，避免重复向量化；
给 Topic 设置了消息过期时间（7 天），避免死信队列堆积过多无效消息。

#### 4. 基于 WebSocket 实现流式对话响应，具体是如何设计的？如何管理会话历史和上下文追踪？在高并发情况下，如何保证 WebSocket 的性能和稳定性？
- 我是用 Spring 的 WebSocket 模块实现的流式对话，具体设计是：
WebSocket 配置：配置了 WebSocket 的握手拦截器，在握手阶段校验用户的 JWT 令牌，解析出用户信息和组织标签，只有鉴权通过的用户才能建立 WebSocket 连接；同时用 SockJS 做降级处理，避免部分浏览器不支持 WebSocket；
会话历史和上下文管理：用 Redis 存储会话信息，每个 WebSocket 连接生成唯一的sessionId，关联用户 ID、组织标签和会话历史；每次对话时，把 Redis 中存储的会话历史拼接成 prompt，和当前问题一起传给 DeepSeek API，实现上下文追踪；
流式响应实现：DeepSeek API 返回流式数据后，通过 WebSocket 的TextMessage逐段推送给前端，前端实时渲染回答内容，实现流式对话的效果；
- 高并发下的性能保证：
用线程池处理 WebSocket 的消息推送，核心线程数设置为 8，最大线程数设置为 16，避免阻塞主线程；
用 Redis 的分布式锁控制同一用户的并发会话，每个用户最多同时建立 2 个 WebSocket 连接，避免资源浪费；
做了心跳检测：每 30 秒发送一次心跳包，超过 2 次未响应则主动关闭连接，清理空闲会话；
压力测试时，模拟 100 并发会话，响应时间稳定在 100ms 以内，没有出现连接超时或消息丢失的情况。

### 技术实现与缓存策略
#### 1. 使用 Redis 实现短信接口调用防护，具体是如何设计的？黑名单机制是如何实现的？如何防止恶意请求？
- 我在项目里是这样设计短信接口防护的：
   频率限制：用 Redis 的incr计数器实现每个手机号的短信发送频率限制 —— 每个手机号作为 Redis 的 key，每次发送短信时incr计数，设置 1 分钟的过期时间，1 分钟内计数超过 3 次就拒绝请求，避免用户频繁刷短信。
   黑名单机制：用 Redis 的Set存储恶意请求的 IP 和手机号，比如连续 10 分钟内发送超过 20 次短信的手机号，或者频繁请求短信接口的 IP，会被加入黑名单，过期时间设为 24 小时；同时用 Spring 的HandlerInterceptor做请求拦截，在短信接口的入口先校验请求的 IP / 手机号是否在黑名单里，在黑名单里直接返回 403。
#### 2. 针对高频访问的商户数据，利用 Redis 实现缓存策略，具体是如何设计的？布隆过滤器是如何避免缓存穿透的？互斥锁和逻辑过期机制是如何解决缓存击穿问题的？
- 我针对商户数据的缓存设计是：
   基础缓存策略：用 Redis 的String存储商户信息，key 是merchant:{id}，value 是序列化后的商户详情（名称、地址、评分等），过期时间设为 30 分钟；更新商户数据时，先更新 MySQL 数据库，再删除 Redis 缓存，避免缓存和数据库不一致。
   布隆过滤器防缓存穿透：用 Redisson 的布隆过滤器实现，把所有商户的 ID 提前存入布隆过滤器，当请求查询商户信息时，先查布隆过滤器，如果过滤器判断该商户 ID 不存在，直接返回 “商户不存在”，避免请求穿透到 MySQL 数据库；之前没加布隆过滤器时，恶意请求查询不存在的商户 ID，导致 MySQL 的 QPS 涨到了 200，加了之后数据库 QPS 直接降到了 80。
   缓存击穿的解决：
   互斥锁：用 Redis 的setnx实现互斥锁，当缓存过期时，只有一个线程能抢到锁，去 MySQL 查询商户信息并更新缓存，其他线程等待 100ms 后重试，适合普通商户数据；
   逻辑过期：针对热门商户（比如评分 4.8 以上的商户），给缓存加一个逻辑过期字段，缓存过期后不直接删除，而是返回旧的商户数据，同时异步开启线程更新缓存，避免热门商户缓存过期时大量请求打向数据库。
#### 3. 你提到通过乐观锁、Redis 分布式锁和缓存预热配合 RabbitMQ 实现高并发下的下单业务，具体是如何实现的？乐观锁和分布式锁分别在什么场景下使用？缓存预热是如何进行的？
我在项目里是这样实现高并发下单的：
流程设计：用户下单→先查 Redis 缓存的商品库存→库存足够则进入下单流程→根据商品类型选择锁机制→扣减库存→发送 RabbitMQ 消息异步处理订单日志和用户积分→返回下单成功。
锁机制的场景选择：
乐观锁：用 MySQL 的version字段实现，下单时 SQL 的where条件加上version = 当前版本，更新成功后version + 1，适合普通商品（库存冲突不频繁的场景），比如日常商品的下单，避免分布式锁的开销；
Redis 分布式锁：用 Redis 的setnx + ex实现（原子操作，避免死锁），下单时先抢锁，抢到锁后再查数据库库存并扣减，适合热点商品（比如优惠券秒杀、限量商品），避免超卖，同时设置锁的过期时间为 10 秒，防止服务宕机导致死锁。
缓存预热：用 RabbitMQ 的延迟队列实现，在商品上线前 1 小时，把热门商品的库存数据提前缓存到 Redis，同时预热商户数据的缓存，避免商品上线时大量请求打向数据库，导致数据库压力过大；比如秒杀商品上线前，提前把库存缓存到 Redis，上线后直接从 Redis 查库存，QPS 能提升 30% 左右。
#### 4. 系统峰值 QPS 提升至 1500，你是如何进行性能测试的？在提升 QPS 的过程中，遇到了什么问题？如何解决的？
性能测试方案：
用 JMeter 做压力测试，模拟 100、200、300 并发的下单和商户查询请求，同时用 VisualVM 监控 JVM 的内存和线程状态，用 Redis 的info命令监控 Redis 的 QPS 和连接数；
测试场景包括普通下单、秒杀下单、商户列表查询，分别统计不同场景下的 QPS 和响应时间。
提升 QPS 的过程中遇到的问题与解决：
问题 1：Redis 连接池过小：一开始 Redis 连接池的最大连接数设为 50，并发请求时出现连接超时，QPS 只能到 800；解决：把 Redis 连接池的最大连接数改成 200，最小空闲数设为 20，同时设置连接超时时间为 500ms，QPS 提升到 1200。
问题 2：数据库慢查询：订单查询时没有加索引，查询时间达到 500ms，拖慢整体 QPS；解决：给订单表的user_id和merchant_id字段加联合索引，查询时间降到 100ms，QPS 提升到 1400。
问题 3：同步处理订单逻辑：下单后同步处理积分更新和订单日志，导致接口响应时间过长；解决：把积分更新和订单逻辑改成用 RabbitMQ 异步处理，接口响应时间从 300ms 降到 100ms，最终 QPS 稳定到 1500。
### 功能实现与性能优化
#### 1. 在订单创建过程中，使用线程池异步处理订单逻辑，具体是如何设计的？线程池的参数是如何配置的？如何避免线程池的溢出问题？
   线程池设计：用 Spring 的ThreadPoolTaskExecutor实现自定义线程池，专门处理订单的非核心逻辑（比如订单日志记录、用户积分更新、下单短信通知），核心逻辑（扣减库存、生成订单号）还是同步处理，避免异步导致的订单一致性问题。
   参数配置：
   核心线程数：8（和服务器 CPU 核数一致，避免上下文切换过多）
   最大线程数：16（核心线程忙不过来时的扩容上限）
   队列容量：200（用 LinkedBlockingQueue 存储待处理任务）
   线程存活时间：60s（空闲线程超过 60s 销毁）
   拒绝策略：CallerRunsPolicy（队列满时，让请求线程自己执行任务，避免直接拒绝请求）
   避免溢出的措施：
   用 Spring Boot Actuator 监控线程池的状态（活跃线程数、队列长度、拒绝次数），如果队列长度超过 150 就告警；
   一开始核心线程数设为 2，导致队列很快满了出现拒绝请求的情况，调整为 8 后解决了这个问题；
   给线程池的任务加了超时控制，用Future.get(5, TimeUnit.SECONDS)避免任务一直阻塞线程池。
#### 2. 基于 Redis 的 Zset 实现点赞排行榜，具体是如何设计的？如何保证点赞数据的实时性？定时任务定期将点赞数据持久化到数据库，具体是如何设计的？定时任务的执行频率是多少？
我在项目里是这样实现点赞排行榜的：
Zset 设计：用 Redis 的 Zset 存储点赞数据，key 是like:rank:daily（按天生成排行榜，避免数据量过大），value 是点评 ID，score 是该点评的点赞数；用户点赞时用ZINCRBY like:rank:daily 1 {comment_id}增加点赞数，取消点赞时用ZINCRBY like:rank:daily -1 {comment_id}减少点赞数。
实时性保证：点赞 / 取消点赞后，通过 WebSocket 把最新的点赞数和排行榜排名推送给前端，前端立刻更新排行榜，实现实时同步；同时用 Redis 的ZRANGE接口，前端可以实时拉取前 20 名的点评排行榜。
持久化定时任务：
用 Spring 的@Scheduled实现定时任务，执行频率是每 30 分钟；
任务逻辑：先获取 Zset 中 score 变化的点评（用ZRANGEBYSCORE筛选出 score 变化的记录），把这些点评的点赞数同步到数据库的comment表的like_count字段，避免全量同步导致 Redis 和数据库压力过大；
同时每天凌晨 0 点，把当天的 Zset 数据备份到 Redis 的持久化库，避免 Redis 宕机导致当天的点赞数据丢失。
#### 3. 系统使用 Nginx，你是如何配置 Nginx 的？Nginx 在这个项目中起到了什么作用？是否进行了负载均衡的配置？
我在项目里是这样配置 Nginx 的：
基础配置：
worker_processes 4（和服务器 CPU 核数一致，充分利用 CPU 资源）
worker_connections 10240（每个 worker 进程的最大连接数）
配置 gzip 压缩：开启 gzip 压缩静态资源（css、js、图片），压缩级别设为 6，减少传输带宽；
配置静态资源缓存：对静态资源设置expires 30d，让浏览器缓存静态资源，减少 Nginx 的请求压力。
负载均衡配置：
在http块里配置upstream，把两个 Tomcat 实例（端口 8080、8081）作为后端服务，用轮询策略做负载均衡，同时配置了健康检查：max_fails=3 fail_timeout=30s，后端服务连续 3 次请求失败就剔除，30 秒后再重试。
Nginx 的作用：
负载均衡：把请求分散到两个 Tomcat 实例，单服务器的 QPS 从 800 提升到 1500；
反向代理：隐藏后端服务的真实地址，提升安全性；
静态资源处理：直接处理静态资源请求，不用转发到 Tomcat，减少 Tomcat 的压力；
跨域配置：在 Nginx 里配置跨域规则，解决前后端跨域问题。
#### 4. 这个项目中，用户登录凭证存储在 Redis 中，结合双重拦截器实现多服务器间 Session 共享，具体是如何实现的？双重拦截器的设计思路是什么？
- 我在项目里是这样实现 Session 共享的：
登录凭证存储：用户登录成功后，生成 JWT 令牌，把 JWT 和用户信息（用户 ID、角色、商户权限）一起存储到 Redis，key 是token:{user_id}，value 是序列化后的用户信息，过期时间设为 30 分钟；用户退出登录时，删除 Redis 中的该 key，实现 JWT 的主动失效。
双重拦截器设计：
第一个拦截器：JWT 校验拦截器：在请求入口拦截所有接口，校验请求头中的 JWT 令牌是否有效（签名是否正确、是否过期），解析出用户 ID 后，从 Redis 中获取用户信息，存入RequestContextHolder，供后续接口使用；如果 JWT 无效，直接返回 401 未授权。
第二个拦截器：Session 共享拦截器：校验 Redis 中是否存在该用户的登录凭证（token:{user_id}），如果不存在，说明用户已退出登录或凭证过期，跳转到登录页；同时校验用户的角色权限，比如普通用户不能访问商户管理接口。

#### 2. 在开发过程中，遇到的最大的难点是什么？如何解决的？
- 开发中最大的难点是高并发秒杀场景下的超卖问题：一开始我用乐观锁（数据库 version 字段）解决超卖，但是在秒杀场景下（比如 1000 人抢 100 张优惠券），乐观锁的冲突率太高，下单成功率只有 30%，而且出现了超卖 10 多单的情况。
解决方法是：
先用Redis 分布式锁做库存预扣减：用户下单前，先抢 Redis 的锁，抢到锁后再查 Redis 缓存的库存，库存足够就扣减 Redis 库存，然后把下单请求发送到 RabbitMQ 异步队列；
用RabbitMQ 异步处理订单：消费者从队列取出请求，再查数据库的库存，用乐观锁扣减数据库库存，避免同步下单导致的数据库压力过大；
加了库存回滚机制：如果数据库扣减失败，把 Redis 的库存加回去，避免 Redis 库存和数据库库存不一致。
#### 3. 你认为这个项目还有哪些可以优化的地方？如果让你重新开发这个项目，你会做哪些改进？
- 如果重新开发这个项目，我会做以下优化：
缓存优化：现在的缓存是被动过期，改成主动缓存更新—— 商户更新商品信息时，主动更新 Redis 缓存，而不是删除缓存；同时用 Redis 的布隆过滤器优化缓存穿透，现在只有商户数据用了布隆过滤器，商品数据还没加，导致恶意请求查不存在的商品 ID 时数据库压力大；
数据库优化：现在的订单表是单表，改成按用户 ID 分表，现在订单表已经有 10 万条数据了，查询用户的历史订单越来越慢，分表后可以提升查询速度；
权限优化：现在的权限是简单的角色判断（用户 / 商户），改成RBAC 权限模型，可以给商户设置不同的权限（比如只能发布商品、可以查看订单），更灵活；
监控优化：现在只有日志记录，加Prometheus+Grafana 监控，监控 Redis 的缓存命中率、数据库的慢查询、RabbitMQ 的队列长度，提前发现问题；
消息队列优化：现在的 RabbitMQ 没有死信队列，改成死信队列 + 重试机制，如果订单处理失败，自动重试 3 次，还失败就通知管理员，避免订单丢失。

# 专业技能
- 讲一下ArrayList、LinkedList、HashMap的底层实现原理？
ArrayList: 底层基于动态数组实现。支持随机访问，查询速度快；但在中间位置插入和删除元素时需要进行内存数据搬移，性能较低。触发扩容时，通常按原容量的 1.5 倍进行增长。

LinkedList: 底层基于双向链表实现。不支持随机访问，查询必须从头或尾遍历，速度较慢；但如果在已知节点进行插入和删除，只需调整指针指向即可，速度极快。

HashMap: 在 JDK 1.8 之后，底层采用“数组 + 链表 + 红黑树”的结构。 当发生哈希冲突时采用链式地址法。为了优化长链表的查询性能，当链表长度大于 8 且数组总容量大于 64 时，链表会转化为红黑树，将查询的时间复杂度从 O(n) 降低到 O(log n)。
### 讲一下都有哪些设计模式？
创建型模式: 关注对象的创建。例如单例模式（保证全局只有一个实例，分为饿汉式和懒汉式，常用于配置类）、工厂模式（解耦对象的创建逻辑与业务逻辑）。

结构型模式: 关注类与对象的组合。例如代理模式（Spring AOP 的核心底层实现，用于在不修改原代码的情况下增强功能）、装饰器模式（动态地给对象添加额外职责）。

行为型模式: 关注对象间的通信。例如策略模式（将一系列算法封装起来并可互相替换，项目中常用来消除冗长的 if-else 分支）、观察者模式（实现事件的发布订阅解耦）。
### 讲一下Redis的内存淘汰策略、数据持久化机制、Redis集群的工作原理？
内存淘汰策略: 当 Redis 内存达到上限时触发。常见的有 noeviction（不淘汰，直接报错）、allkeys-lru（淘汰全局最近最少使用的键）、volatile-lru（只在设置了过期时间的键中淘汰最近最少使用的）、allkeys-random（随机淘汰）等。

数据持久化机制:

RDB (快照): 在指定时间间隔内生成内存数据的二进制文件。恢复速度快，但意外宕机会丢失最后一次快照后的数据。

AOF (追加日志): 将每次写操作记录到日志文件中。数据安全性更高，但文件体积较大，重启恢复速度比 RDB 慢。

Redis 集群工作原理:  采用哈希槽（Hash Slot）机制，将数据均匀映射到 16384 个槽位中，由多个主从节点分担压力。节点间通过无中心化的 Gossip 协议进行通信，支持自动的故障转移和动态扩容缩容。
### 高并发下缓存穿透、雪崩、击穿是如何解决的？
缓存穿透: 恶意请求查询数据库中根本不存在的数据，导致请求直接穿透 Redis 打穿数据库。解决：使用布隆过滤器拦截不存在的 Key，或者对查询为空的结果也设置一个短期的空缓存。

缓存雪崩: 大量缓存数据在同一时间集中过期，或者 Redis 节点直接宕机，引发数据库压力剧增。解决：给缓存过期时间加上随机值避免同时失效；搭建 Redis 高可用集群；在业务层引入限流降级机制。

缓存击穿: 某个极度高频的热点 Key 突然过期，导致瞬间海量请求直奔数据库重建缓存。解决：使用互斥锁（如 Redisson 分布式锁，只放行一个线程去查库重建，其他线程等待），或者设置热点数据逻辑过期不停机更新。
### 讲一下怎么优化的慢SQL
  排查定位: 开启 MySQL 的慢查询日志捕获慢 SQL。使用 EXPLAIN 关键字分析执行计划，重点关注 type（访问类型是否全表扫描）和 Extra（是否使用了文件排序 filesort）。

索引优化: 严格遵循最左前缀法则。避免索引失效的场景，如：在索引列上做计算或使用函数、隐式类型转换、使用前导模糊查询（%like）。

语句与架构优化: 坚决避免 SELECT *，尽量使用覆盖索引减少回表操作。深分页场景下使用延迟关联优化。针对超大表，考虑分库分表或将历史数据归档。
### 讲一下事务、MVCC、索引结构、锁机制的核心原理
  事务 (ACID): 依赖底层日志实现。原子性由 Undo log 保障；持久性由 Redo log 保障；隔离性由锁和 MVCC 保障；最终实现一致性。

MVCC (多版本并发控制):  核心由“记录的隐藏字段 + Undo log 版本链 + Read View (读视图)”组成。它让读写操作互不阻塞，在“可重复读”隔离级别下，完美解决了不可重复读的问题。

索引结构: InnoDB 默认采用 B+ 树。 它的非叶子节点只存储索引键不存数据，所有的行数据（或主键值）统一挂在叶子节点，并通过双向链表连接。这种极矮胖的结构极大减少了磁盘 I/O 次数，并支持高效的范围查询。

锁机制: 包含全局锁、表级锁和行级锁。行锁又细分为记录锁（Record Lock，锁单行）、间隙锁（Gap Lock，锁区间防插入）和临键锁（Next-Key Lock，前两者结合，有效防止幻读）。
### Spring Bean的生命周期？怎么实现的自动配置管理？Spring都有哪些配置？
  Bean 生命周期:  核心经历四个阶段：实例化 (Instantiation，分配内存) -> 属性填充 (Populate Properties，依赖注入) -> 初始化 (Initialization，执行 init-method 等方法) -> 销毁 (Destruction)。

自动配置管理: 核心在于 @SpringBootApplication 注解内部的 @EnableAutoConfiguration。应用启动时，它会扫描第三方 Jar 包下的 META-INF/spring.factories 或 AutoConfiguration.imports 文件，加载配置类。然后通过 @Conditional 系列注解（如根据 classpath 下是否存在某个类）来进行条件判断，按需将 Bean 注入到 Spring 容器中，实现“开箱即用”。

Spring 常见配置方式: 早期使用的 XML 配置文件；基于组件扫描的注解配置（如 @Service, @Autowired）；目前最推崇的 Java 配置类（使用 @Configuration 和 @Bean 完全替代 XML）。
